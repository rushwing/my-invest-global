# Goal

提供 CLI 入口供 cron/人工触发每日 pipeline，并通过端到端集成测试验证 `run_daily_pipeline()` 在4种关键场景下的正确性。

# In Scope

- **`scripts/run_pipeline.py`**（新建）：
  - argparse 解析：`--date YYYY-MM-DD`（必填）；`--synthesize`（可选 flag，Phase 6 实现，此 REQ 仅解析不执行）
  - 日期格式验证：不合法格式打印错误并 `sys.exit(1)`
  - 实例化 `StubSignalProvider`（默认）；Phase 2–4 完成后在此处替换为真实 providers
  - 调用 `run_daily_pipeline(date, tech, fund, sent)`
  - 成功后打印 `SignalsFile` 写出路径
  - 失败时打印错误并以非零退出码退出

- **`tests/test_runner.py`**（新建）：
  - 使用 in-memory DuckDB fixture（`:memory:`）+ 最小化 CSV holdings fixture
  - 4个集成测试场景（见 § Test Cases）
  - 测试中显式传入 `StubSignalProvider(base_score=...)` 以控制 composite_score
  - 使用 `tmp_path`（pytest fixture）隔离文件系统写出，不污染 `data/processed/`
  - 测试结束后验证输出文件通过 `SignalsFile(**json.load(...))` Pydantic 校验

# Out of Scope

- `--synthesize` 的实际执行（Phase 6 负责，此 REQ 仅 argparse 解析，不调用 Claude API）
- Streamlit 集成测试（Phase 7）
- 性能/压力测试

# Acceptance Criteria

- `uv run python scripts/run_pipeline.py --date 2026-05-13` 对 fixture 数据写出 `signals_20260513.json`，文件通过 `SignalsFile` Pydantic 校验
- `uv run python scripts/run_pipeline.py --date BADDATE` 以非零退出码退出，不写出任何文件
- `uv run python scripts/run_pipeline.py --date 2026-05-13 --synthesize` 不报错（仅 flag 解析，不执行 Claude API）
- `uv run pytest tests/test_runner.py -v` 全部通过，覆盖以下4个场景：
  1. normal path：composite=70，action_code='hold_add'
  2. stop_loss override：价跌8%，action_code='stop_loss'
  3. macro_red downgrade：macro_state=RED + elastic 股 + stub76，action_code='hold_add'（从 strong_add 降级）
  4. portfolio rebalance 触发：elastic_ratio=43%，rebalance_needed=True，direction='reduce_elastic'

# Test Cases

需覆盖以下场景（TC文件由 evaluator-001 在 `tc_design` 阶段编写）：

1. **TC-005-01 normal path**：
   - Fixture：两只白马股+一只弹性股，`StubSignalProvider(70.0)`
   - 预期：写出 JSON，`composite_score=70.0`，`action_code='hold_add'`，`rebalance_needed=False`
   - 验证：`SignalsFile(**data)` 不抛异常

2. **TC-005-02 stop_loss override**：
   - Fixture：一只弹性股，收盘价 = 买入成本 × 0.91，`StubSignalProvider(75.0)`（原本 strong_add）
   - 预期：`action_code='stop_loss'`（composite 高但止损覆盖）

3. **TC-005-03 macro_red downgrade**：
   - Fixture：macro_state=RED，一只弹性股，`StubSignalProvider(76.0)`（原本 strong_add）
   - 预期：`action_code='hold_add'`（降一级）
   - 验证：白马股不受 macro_red 降级影响

4. **TC-005-04 portfolio rebalance 触发**：
   - Fixture：弹性股总市值 / 总市值 = 0.43（43%，偏离 YELLOW 目标 33% 超过10%）
   - 预期：`SignalsFile.portfolio_summary.rebalance_needed=True`，`direction='reduce_elastic'`

# Implementation Notes

- `scripts/run_pipeline.py` 中 `StubSignalProvider` 仅用于 Phase 5 阶段；Phase 2–4 完成后替换为真实 providers，此文件是唯一需要修改的接入点
- 日期验证使用 `datetime.strptime(date, '%Y-%m-%d')` 捕获 `ValueError`
- `tests/test_runner.py` 中的 in-memory DuckDB fixture 需预先创建必要表（`stock_quotes`、`stock_meta` 最小化）并插入 fixture 数据
- 文件写出路径用 `tmp_path / f"signals_{date}.json"` 覆盖默认路径（通过 monkeypatch 或 runner 参数传入 output_dir）
- 注意：`run_daily_pipeline()` 需支持可选的 `output_dir: Path = Path("data/processed")` 参数以便测试注入；若 REQ-004 未设计此参数，在本 REQ 的 req_impl 阶段向 runner 添加（属于 REQ-004 的扩展，需在 REQ-004 中补充说明）
