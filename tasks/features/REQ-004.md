# Goal

实现 `run_daily_pipeline()`——Phase 5 的核心编排器。它以依赖注入方式接受三个 `SignalProvider` 实现，对每只持仓股并行计算信号并融合，应用特殊覆盖规则，写出 `SignalsFile` JSON。

# In Scope

- **`engine/runner.py`**（新建）：
  - `run_daily_pipeline(date: str, technical: SignalProvider, fundamental: SignalProvider, sentiment: SignalProvider) -> SignalsFile`
  - 调度顺序（严格按 PHASE-005.md 规格）：
    1. 从 DuckDB 加载当日 holdings（`Storage` → `SELECT * FROM stock_meta WHERE code IN (...)` + HoldingsFile CSV）
    2. `ThreadPoolExecutor(max_workers=5)` 并行对每只持仓股计算三信号（每股3次 `compute_score()` 调用）
    3. `composite_score = technical_score × 0.30 + fundamental_score × 0.40 + sentiment_score × 0.30`（精度保留2位小数）
    4. 映射基础 `action_code`（≥75 → `strong_add`；60–74 → `hold_add`；40–59 → `hold`；<40 → `reduce`）
    5. 应用特殊覆盖（按优先级从高到低）：
       - **stop_loss**：当日收盘价 < 买入成本价 × 0.92 → `action_code = 'stop_loss'`（最高优先级）
       - **macro_red**：`macro_state == RED` AND `category == 'elastic'` → `action_code` 降一级（`strong_add→hold_add`，`hold_add→hold`，`hold→reduce`）；若已为 `stop_loss` 则不再降级
       - **take_profit_alert**：当日收盘价 > 买入成本价 × 1.20 → 在 `StockSignal.signals` dict 中增加 `"take_profit_alert": true`（不替换 `action_code`）
    6. 调用 `check_portfolio_balance(holdings_df, macro_state)` 获取再平衡状态
    7. 调用 `check_position_too_large(holdings_df)` 获取超限弹性股列表
    8. 构造 `SignalsFile` Pydantic 对象并验证
    9. 写出 `data/processed/signals_{date}.json`（幂等：同日期覆盖）

- **收盘价/买入成本**：从 DuckDB `stock_quotes` 表读取当日收盘价（`close`字段）；买入成本从 HoldingsFile `cost_price` 列读取

- **macro_state 获取**：在函数开始时调用 `get_macro_state()`（来自 REQ-001）

# Out of Scope

- Claude API 调用（Phase 6）
- Streamlit 展示（Phase 7）
- 实际交易执行（永不实现）
- 真实 `technical_score` / `fundamental_score` / `sentiment_score` 实现（Phase 2–4 各自实现后替换注入的 provider）

# Acceptance Criteria

- `run_daily_pipeline('2026-05-13', stub70, stub70, stub70)` 对 in-memory fixture：
  - 写出 `data/processed/signals_20260513.json`（文件存在）
  - `composite_score == 70.0`，`action_code == 'hold_add'`（60≤70<75）
  - 返回的 `SignalsFile` 通过 Pydantic schema 验证（`SignalsFile(**json.load(...))` 不抛异常）
- stop_loss 覆盖：当日价 = 买入成本 × 0.91 → `action_code == 'stop_loss'`（不受 composite_score 影响）
- macro_red 覆盖：`macro_state=RED` + 弹性股 + `composite_score=75` → `action_code == 'hold_add'`（从 `strong_add` 降一级）
- take_profit_alert：当日价 = 买入成本 × 1.25 → `signals["take_profit_alert"] == True`，基础 `action_code` 不变
- 同日期重复运行不报错（幂等）
- `ThreadPoolExecutor(max_workers=5)` 限制并发：fixture 中6只股票的调用总次数 = 18（6×3），耗时不超过2秒
- `uv run mypy engine/runner.py --strict` 通过

# Implementation Notes

- `SignalProvider` 类型注解引用 `engine.signals.protocol.SignalProvider`（REQ-003）
- Holdings 加载：优先从 DuckDB `stock_quotes` / `stock_meta` 读取；若 DuckDB 中无数据（空数据库），从 `HoldingsFile` CSV 读取并用作 fallback（便于测试时使用 in-memory DuckDB + CSV fixture）
- 测试使用 `Storage(":memory:")` 创建 in-memory DuckDB，加载最小 fixture 数据
- `data/processed/` 目录写出前确保存在（`Path.mkdir(parents=True, exist_ok=True)`）
- stop_loss 边界：严格小于（`price < cost × 0.92`），等于不触发
- `composite_score` 计算使用 `round(result, 2)` 保留两位小数
- 函数签名中 `technical`/`fundamental`/`sentiment` 参数不指定默认值，调用方必须显式传入（强制依赖注入，避免意外使用 stub）
